The Missing Contract

AI systems have become deeply phase-structured while the memory control plane remains largely phase-blind. Prefill pushes bandwidth. Decode punishes latency. Agentic execution injects bursty, variance-heavy traffic. Training oscillates between forward-read pressure and backward-write pressure. We have rich software semantics above the kernel and highly tuned signaling machinery below it, but almost no narrow interface joining the two.

`/dev/mem_hint` is the shape of that missing contract. It is small on purpose: an eight-byte hint packet, a kernel policy engine, a safety limiter that software cannot outrank, and a set of illustrative transport paths into a memory controller or CXL-attached device.

Why This Lives in the Kernel

The kernel sits at the point where privilege, scheduling context, telemetry, and hardware access intersect. A runtime can describe intent in user space, but someone still has to validate that intent, arbitrate between explicit and inferred signals, expose operating policy, and map the request to a privileged delivery path. That makes the Linux kernel a natural control-plane host even if final safety enforcement belongs in immutable hardware.

Three Entry Paths

  • Explicit writes from inference or training runtimes when software knows phase boundaries exactly.
  • PMU-driven auto-classification for situations where runtime patches are unavailable.
  • sysfs tuning for operators who need live control over thresholds and per-phase defaults.

Phase Semantics Matter

The important shift is not that software can "tune memory." Firmware and board code already do that. The shift is that the tuning can become workload-aware in a way that tracks AI execution phases instead of static worst-case assumptions. Decode is not prefill. Backward is not forward. Agentic tool-use is not steady-state token generation. 

phase_id | meaning
---------+-------------------------------------------
0x01     | Prefill
0x02     | Decode
0x03     | Agentic / tool-oriented
0x04     | Idle
0x05     | Training forward pass
0x06     | Training backward pass

Safety Is Not Optional

Any system that lets software influence timing or signaling must make safety the first-class story, not a footnote. That is why the reference architecture keeps a hard distinction between policy and enforcement. The driver proposes. The safety limiter decides. In a real platform, the limiter would be hardware logic bounded by JEDEC requirements, ECC telemetry, and security-state floors.

CXL Changes the Shape of the Problem

Once memory extends across a CXL fabric, the question is no longer only "what timing should this DDR channel use?" It becomes "which region should respond how, under what congestion state, and with what locality?" A meaningful embodiment of workload-aware memory policy eventually has to reason about fabrics, HDM regions, and NUMA-aware propagation.
